## ЮKassa

### Структура отчёта по зачислениям

Коннектор `~~платежи` (ЮKassa) отдаёт отчёт «История операций» или «Реестр
платежей». Ключевые поля:

| Поле | Примечания |
|---|---|
| `payment_id` | Идентификатор платежа в ЮKassa |
| `created_at` | Когда прошла операция |
| `amount.value` | Валовая сумма (положительная = деньги пришли) |
| `income_amount.value` | Сумма за вычетом комиссии ЮKassa (чистое зачисление) |
| `status` | Бери только `succeeded` (успешно) и `pending` (в ожидании) |
| `metadata` / `description` | Привязка к заказу/клиенту — полезно для сопоставления |

**Дата зачисления vs дата операции:** ЮKassa перечисляет деньги пакетами.
Продажа 28 апреля может зачислиться на расчётный счёт только 2 мая. Сопоставляй
по **дате зачисления** при сверке с поступлениями в учёте, а не по дате операции.

**Возвраты:** Проходят отрицательными суммами (`refund`). Они должны гасить
исходную продажу — не помечай возврат расхождением против учёта, если в учёте
есть соответствующий корректировочный документ.

**Заблокированные средства:** Операции в статусе `waiting_for_capture` или
зависшие в холде ещё не зачислены. Отметь их отдельно — попадут в зачисления
следующего месяца.

## Тинькофф Касса

### Структура отчёта по зачислениям

Тинькофф называет это **реестром платежей / выгрузкой по эквайрингу**. Каждое
зачисление покрывает 1–2 рабочих дня продаж.

Ключевые поля выгрузки:

| Поле | Примечания |
|---|---|
| `PaymentId` | Идентификатор платежа |
| `SettlementDate` | Дата зачисления средств на счёт |
| `Amount` | Чистое зачисление (уже минус комиссия Тинькофф), в копейках |
| `Status` | Для сверки бери только `CONFIRMED` |

**Комиссия:** Тинькофф удерживает комиссию до зачисления — значит `Amount` уже
чистый. При сверке с учётом сравнивай с чистым поступлением, а не с валовой
суммой продажи.

**Зачисление день в день:** Если у собственника подключено мгновенное
зачисление, средства могут прийти в дату операции. По умолчанию — следующий
рабочий день.

## Банк-выписка (СБП / расчётный счёт)

### Структура выписки

Выписка из банк-клиента (1С-совместимая выгрузка или CSV) — основной источник,
когда эквайринг не подключён или платежи идут через СБП напрямую на счёт.

Ключевые поля:

| Поле | Примечания |
|---|---|
| `Дата проводки` | Дата зачисления/списания по счёту |
| `Сумма` | Сумма по дебету (приход) или кредиту (расход) |
| `Назначение платежа` | Текст основания — для привязки к счёту/клиенту |
| `ИНН плательщика` | Идентификация контрагента |

**СБП:** Поступления по СБП обычно приходят на счёт почти моментально и уже
чистыми (комиссия эквайера СБП удерживается отдельным списанием — учитывай её,
если она проходит отдельной строкой).

**Эквайринг общим платежом:** Банк может зачислять выручку эквайринга одной
суммой за день/период — тогда одно поступление в выписке соответствует пачке
операций в отчёте эквайринга. Сопоставляй пачку, а не строку-в-строку.

## Логика сверки (все источники)

Используй эту логику для сопоставления зачислений с поступлениями в учёте:

```
для каждого зачисления за закрываемый месяц:
    найди поступление в учёте, где:
        abs(учёт.сумма - зачисление.чистая_сумма) < 0,50 ₽
        И abs(учёт.дата - зачисление.дата_поступления) <= 2 дня

    если совпадение найдено:
        отметь СВЕРЕНО
    иначе если abs(учёт.сумма - зачисление.чистая_сумма) < 0,50 ₽ (расходятся только даты):
        пометь РАСХОЖДЕНИЕ_ДАТ (обычно разница в сроках — низкий приоритет)
    иначе если зачисление не сопоставлено вообще:
        пометь НЕТ_В_УЧЁТЕ
    иначе если поступление в учёте не сопоставлено ни с одним зачислением:
        пометь ПОСТУПЛЕНИЕ_БЕЗ_ПАРЫ (может быть наличные, перевод по реквизитам или ошибка статьи)
```

**Несколько источников платежей:** Прогоняй логику отдельно по каждому источнику
(`~~платежи`), затем агрегируй флаги. Поступление в учёте, не совпавшее с ЮKassa,
может законно оказаться зачислением Тинькофф или СБП — не помечай его, пока не
проверил все подключённые источники.
