# Сверка с данными бухгалтерии (1С и др.) — справочник

> Имя файла оставлено прежним (`quickbooks-reconcile.md`), чтобы не ломать
> ссылки из SKILL.md. Содержание — сверка по данным учёта в 1С / МойСклад.

## Какие отчёты подтянуть

### Отчёт о прибылях и убытках (ОПиУ)

Используй отчёт **«Прибыли и убытки» / финрезультат** через коннектор
`~~бухгалтерия` (1С, МойСклад):
- Период: с первого по последний день закрываемого месяца
- Метод учёта: **начисление**, если собственник не сказал, что ведёт по оплате
  (кассовый метод)
- Включай все статьи

Ключевые поля для дальнейшей работы:
| Поле | Примечания |
|---|---|
| `Итого выручка` | Выручка верхней строкой |
| `Валовая прибыль` | Выручка минус себестоимость |
| `Валовая маржа` | Считай как Валовая прибыль / Итого выручка |
| `Чистая прибыль` | Итог |
| `Итого расходы` | Подытог операционных расходов |

### Реестр операций

Подтяни отчёт **«Список операций по датам»** или эквивалент за закрываемый
месяц. Это построчная детализация для пометки операций без статьи и поиска
дублей.

Ключевые поля:
| Поле | Примечания |
|---|---|
| `Дата документа` | Дата операции (не дата проводки) |
| `Вид операции` | Счёт, Оплата, Расход, Поступление и т.д. |
| `Сумма` | Положительная = доход, отрицательная = расход в большинстве выгрузок |
| `Статья` | Статья доходов/расходов в учёте |
| `Контрагент` | Поставщик или клиент |
| `Назначение` | Свободное описание |
| `Документов приложено` | > 0 означает, что первичка прикреплена в учёте |

## Как определить операции без статьи

Помечай операцию как «без статьи», если поле `Статья` одно из:
- «Без статьи» / «Не распределено»
- «Прочие доходы» / «Прочие расходы» (когда используется как свалка)
- пусто / null
- «Уточнить у бухгалтера» (встроенная категория «не знаю»)

## Пагинация

Выгрузки из учёта могут возвращать максимум 1 000 строк за запрос. Для бизнесов
с большим объёмом операций разбивай постранично (`startPosition` и `maxResults`
или аналог). Запрашивай пачками по 500 строк, чтобы надёжно держаться под
лимитом.

## Частые проблемы с данными

**Разбитые операции** — одна покупка, разнесённая по нескольким статьям,
появляется несколькими строками с той же датой и контрагентом, но разными
суммами. Это НЕ дубли. Перед пометкой дублей группируй строки по
`номер_документа` — строки с общим номером это разбивки одной операции.

**Банковский импорт vs ручной ввод** — операции из импорта банк-выписки имеют
в назначении/примечании метку «банк-клиент» или похожую. Ручной ввод часто без
примечания. Само по себе ни то ни другое не сигнал проблемы, но помогает понять,
почему у операции может не быть первички.

**Операции эквайринга, уже попавшие в учёт** — если у собственника настроена
автозагрузка из `~~платежи` (ЮKassa/Тинькофф) в учёт, операции эквайринга уже
будут в реестре. Не считай их дважды при сверке (Шаг 3). Проверь, есть ли в
поступлениях в учёте метка эквайринга в статье или назначении, прежде чем
сопоставлять.

**Авансы / предоплаты** — предоплата клиента не выручка, пока работа не сдана.
Если собственник ведёт учёт по оплате (кассовый метод), различие менее важно,
но всё равно помечай крупные операции вида «Поступление/Аванс» без сопоставленного
счёта-реализации — чтобы собственник о них знал. На УСН доход признаётся по
оплате, поэтому аванс попадает в налоговую базу сразу — отметь это собственнику.
