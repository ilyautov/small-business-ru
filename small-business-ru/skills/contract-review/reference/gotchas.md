# Пограничные случаи

Тонкие места анализа договоров. Пары «Плохо / Хорошо».

---

## Случай: флагуем стандартный оборот как red flag

**Почему важно:** избыточное флагование приучает владельца игнорировать сводку. Если всё — red flag, значит ничего не red flag.

### ✗ Плохо

```
В договоре стандартный взаимный пункт о конфиденциальности, срок 2 года.
Claude: «🔴 Red flag: пункт о конфиденциальности накладывает обязательства на обе стороны».
```

Взаимная конфиденциальность — рыночная норма. Флагуя её, вы убиваете соотношение сигнал/шум.

### ✓ Хорошо

```
В договоре стандартный взаимный пункт о конфиденциальности, срок 2 года.
Claude: [пропускает; не флагует]
```

Резервируйте red flags для действительно нестандартных условий. Если пункт типовой и честный — опустите.

---

## Случай: отсутствующие пункты — не отсутствуют в анализе

**Почему важно:** для малого бизнеса отсутствие стандартных защит (ограничение ответственности, порядок изменения объёма, исключение упущенной выгоды) часто опаснее невыгодного пункта — пробел заполнит ГК РФ, обычно в пользу контрагента.

### ✗ Плохо

```
В договоре нет ограничения ответственности.
Claude: [не упоминает; анализирует только то, что есть в тексте]
```

Владелец подписывает, думая, что ответственность ограничена. Это не так.

### ✓ Хорошо

```
В договоре нет ограничения ответственности.
Claude: «🔴 Отсутствует: ограничение ответственности. В договоре нет пункта,
ограничивающего размер ответственности. По умолчанию (ГК РФ) ваша ответственность,
включая упущенную выгоду, не ограничена. Рыночная практика — ограничить суммой
оплаты по договору за последние 12 месяцев. Предложите добавить: "Общая
ответственность каждой стороны не превышает суммы, уплаченной по Договору за
12 месяцев, предшествующих предъявлению требования; упущенная выгода возмещению
не подлежит."»
```

Считайте отсутствующие-но-стандартные пункты red flags. Явно помечайте «Отсутствует».

## Случай: большие PDF обрезаются на середине анализа

**Почему важно:** в договорах на 20–30+ страниц самые опасные условия часто в приложениях, спецификациях, доп. соглашениях в конце. Если PDF прочитан только до 15-й страницы, ключевые условия молча пропущены.

### ✗ Плохо

```
Скилл читает страницы 1–10 договора поставки на 40 страниц. Спецификации А–Г
(с 28-й страницы) содержат порядок передачи прав и неустойку. Скилл выдаёт «чистую» сводку.
```

### ✓ Хорошо

```
Читайте PDF частями: страницы 1–10, 11–20, 21–40. Проанализируйте все части
перед сводкой. Если заголовок раздела упоминает «Приложение», «Спецификация»
или «Дополнительное соглашение» — явно отметьте и убедитесь, что он прочитан.
```

Всегда читайте документ целиком. Используйте параметр `pages` для разбивки больших PDF.

## Случай: типовой шаблон контрагента vs. согласованный проект

**Почему важно:** рекомендации должны соответствовать переговорной ситуации. Жёстко давить на первый проект договора небольшого стартапа — не то же самое, что давить на закупочный шаблон крупного холдинга (поставщик Газпром/Сбер/Росатом): последний часто непреклонен по 80% условий.

### ✗ Плохо

```
Закупочный договор крупного холдинга со стандартными условиями.
Claude: «🔴 Эту отсрочку 60 дней нужно пересогласовать на постоплату по факту.
         Их юристы наверняка примут правку».
```

Нереалистичная рекомендация тратит переговорный капитал владельца впустую.

### ✓ Хорошо

```
Claude: «🟡 Отсрочка оплаты 60 дней. Это дольше типичного (постоплата по факту
         или 30 дней — норма), но обычно для закупок крупных холдингов. Стоит
         попросить отсрочку 45 дней как компромисс — могут согласиться, особенно
         для повторных поставок. Учтите: если это закупочный шаблон крупного
         заказчика, условия оплаты часто непереговариваемы в первом договоре».
```

Соотносите рекомендацию с расстановкой сил. Помечайте жёлтым, не красным. Признавайте реальность.

## Случай: неясный статус НДС у УСН-контрагента

**Почему важно:** с 2026 порог освобождения УСН от НДС снижен до 20 млн ₽. Контрагент на УСН, перешагнувший порог, становится плательщиком НДС автоматически — и цена «без НДС» в старом договоре превращается в спор о том, кто доплачивает налог.

### ✗ Плохо

```
В договоре цена «1 000 000 рублей» без указания статуса НДС.
Claude: [не флагует; считает цену понятной]
```

### ✓ Хорошо

```
Claude: «🔴 Неясный статус НДС. Цена указана как "1 000 000 рублей" без оговорки
         про НДС. С 2026 контрагент на УСН с доходом свыше 20 млн ₽ становится
         плательщиком НДС. Если он перешагнёт порог, возникнет спор: входит налог
         в эту сумму или начисляется сверху. Зафиксируйте явно: "в том числе
         НДС __%" / "без НДС" / "НДС не облагается на основании п. 2 ст. 346.11
         НК РФ (УСН)". Сверьтесь с бухгалтером по фактическому статусу сторон».
```

Всегда требуйте явную формулировку статуса НДС в цене договора.
