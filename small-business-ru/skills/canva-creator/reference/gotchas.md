# Подводные камни

Типичные режимы сбоя скилла canva-creator.

> Во всём файле под `~~дизайн` понимается сервис дизайна (VistaCreate, Supa,
> Поликрафт), под `~~crm` — сервис постановки публикаций в очередь
> (Битрикс24, amoCRM). Каналы РФ: Telegram, VK, Дзен, Одноклассники, RuTube.

---

## Камень: генерация ЛЮБОГО макета `~~дизайна` для email-строки

**Почему важно:** владелец полностью вывел `~~дизайн` из email-пути.
Автозаполнение email-шаблонов даёт заглушки-картинки (стоковые пейзажи,
пустые прямоугольники) в плитках, а превью вариантов не рендерятся в чате.
Решение было: письма из этого скилла — только текст, точка. Сгенерировать
email-макет — даже один — это откат явного решения.

### ✗ Плохо

```
В календаре 6 строк соцсетей + 2 email-строки.
→ Стадия 2 генерирует 6 соцмакетов + 2 email-макета.
→ Email-макеты выходят с плитками-заглушками.
```

Или любой вариант: «сгенерирую email-шаблон просто для референса», «нашёл email-шаблон, показать превью?», «вот макет для письма 05.06…» — всё запрещено.

### ✓ Хорошо

```
Стадия 1: календарь собран со столбцом «Путь».
  - 6 строк помечены «Макет (соцсети)»
  - 2 строки помечены «Только текст» (без ~~дизайна)
Стадия 2: генерирует ровно 6 макетов ~~дизайна. Email-строки не передаются
          ни в один эндпоинт ~~дизайна.
Стадия 3: пишет 6 подписей для соцсетей + 2 полных письма (тема + прехедер + тело) текстом.
Стадия 5: ставит 6 соцпостов в очередь ~~crm. Выдаёт 2 письма инлайн для копирования в сервис рассылки.
```

Если владелец явно просит email-макет в `~~дизайне` («можешь сделать версию письма 05.06 в VistaCreate?»), перенаправьте: «Этот скилл держит письма текстовыми, потому что email-путь в дизайн-сервисе даёт заглушки-картинки. Если хотите письмо с дизайном — соберите его прямо в `~~дизайне`, а я помогу с текстом здесь.»

## Камень: генерация без плана бюджета

**Почему важно:** `~~дизайн` ограничивает частоту запросов, и каждый макет
стоит ~5 вызовов (автозаполнение + экспорт + опрос). Кампания на 10 строк с
4 кандидатами на строку генерирует 40 макетов и ~200 вызовов API — а при
параллелизме хуже. Без предполётного бюджета владелец упирается в квоту
посреди кампании без понятного пути восстановления, и четыре уже
сгенерированных поста лежат без дела.

### ✗ Плохо

```
Календарь согласован → сразу запускаем 4 кандидата × 8 строк параллельно
→ ~160 вызовов за 90 секунд → RATE_LIMIT_EXCEEDED на 5-м посте → цикл
повторов → владелец заблокирован.
```

### ✓ Хорошо

```
Предполёт: «8 строк под макет × 3 кандидата = 24 макета, ~120 вызовов API.
Идём по умолчанию 3 кандидата на строку?»

Стадия 3: 3 кандидата строки 1 параллельно → пауза 30 с → строка 2 →
пауза 30 с → строка 3 → … заканчивается на ~12 макетах/минуту, с запасом.

Если квота всё же сработала: стоп, показать прогресс, спросить владельца —
перейти на одного кандидата, пауза на час, или к текстам с тем, что есть.
```

## Камень: пропуск Контрольной точки 1 (согласование календаря) до генерации

**Почему важно:** генерация 10+ макетов `~~дизайна` стоит вызовов API,
времени и внимания. Если в календаре не те даты, не те каналы или владелец
передумал насчёт темы — вся работа в мусор. Контрольная точка календаря —
самое дешёвое место поймать рассинхрон.

### ✗ Плохо

Получили бриф → сразу начали вызывать автозаполнение по всем 12 постам → показываем макеты до того, как владелец согласовал расписание.

Владелец: «Стоп, я не хотел постов на неделе с 15-го, я в отъезде.»

### ✓ Хорошо

Сначала показать таблицу-календарь. Дождаться явного «годится», прежде чем открывать хоть один вызов API `~~дизайна`.

## Камень: в шаблоне больше слотов под изображения, чем фото в брифе

**Почему важно:** многослотовые соцшаблоны (карусели, товарные сетки) могут
иметь 3+ слота. Когда бриф даёт 1 фото, а шаблон требует 3, незаполненные
слоты молча рендерят стоковую заглушку — облака и зелёные холмы вместо фото
товара. Макет выглядит «готовым», пока владелец его не откроет.

(Примечание: email-строки полностью пропускают `~~дизайн` — см. SKILL.md, Стадия 3. Этот камень касается только соцшаблонов с несколькими слотами.)

### ✗ Плохо

Собрать манифест с одной строкой на макет («Карусель — нужен Hero_Image»). Загрузить одно фото. Сгенерировать. Карусель выходит со слайдом 1 заполненным и слайдами 2–3 с одинаковыми стоковыми пейзажами.

### ✓ Хорошо

Собрать манифест по слотам. Показать пробел явно:

```
В шаблоне «карусель из 3 товаров» 3 слота под изображения. Бриф дал мне
1 фото. Чем заполнить остальные 2?

1. Повторить то же фото во всех 3 слотах
2. Я пришлю ещё 2 фото
3. Выбрать одно-слотовый шаблон
```

Дождаться выбора владельца до генерации. Не дать заглушкам уйти в публикацию.

## Камень: доверять `status: success` без экспорта и визуальной проверки

**Почему важно:** успешная задача автозаполнения значит, что `~~дизайн`
принял запрос и создал дизайн — не что в дизайне правильные изображения.
Если `asset_id` был пустым, устаревшим или не того типа, задача всё равно
вернёт `status: success`, а дизайн отрендерится с дефолтными заглушками.
Единственный способ узнать — посмотреть на готовый вывод.

### ✗ Плохо

```
Статус задачи: success ✓
result.design.id: есть ✓
→ Показать владельцу
```

Результат: владелец открывает карусель и видит стоковые пейзажи в 3 из 4 макетов.

### ✓ Хорошо

После успеха задачи экспортируйте каждый дизайн в постоянный PNG. Посмотрите на каждый PNG и подтвердите, что в нём реальные ассеты — не дефолтный пейзаж, не серые прямоугольники, не текст-рыба. Показывайте владельцу только после прохождения визуальной проверки.

---

## Камень: один макет в пакете не отрендерился

**Почему важно:** когда 1 из 4 макетов в пакете сбоит (ошибка задачи,
истёкший URL превью, отрендеренная заглушка), а вы показываете частичную
карусель, владелец видит три реальных макета и один серый прямоугольник
«Показать изображение». Он теряет доверие к остальным, хотя они в порядке.

### ✗ Плохо

```
Макеты недели 1 готовы:
[Вариант A — реальное превью]
[Вариант B — реальное превью]
[Вариант C — реальное превью]
[Вариант D — заглушка «Показать изображение»]
Какой использовать?
```

### ✓ Хорошо

Когда поймали битый макет на проверке (Шаг 6), перегенерируйте только его с правильным asset_id. Переэкспортируйте, перепроверьте. Показывайте только когда все четыре макета пакета имеют постоянные проверенные превью. Если вторая попытка тоже сбоит — спросите владельца: пропустить, сменить шаблон или попробовать другое фото.

---

## Камень: генерация без подтверждения, что у всех полей автозаполнения есть значения

**Почему важно:** автозаполнение молча рендерит пустые заглушки, когда
значение поля отсутствует или указанный `asset_id` не существует. Результат —
профессионально выглядящий шаблон с пустыми белыми прямоугольниками там, где
должны быть фото товара и заголовок.

### ✗ Плохо

Собрать календарь → сразу запустить всё автозаполнение параллельно → половина макетов приходит с пустыми заглушками, потому что фото товаров не были загружены заранее.

### ✓ Хорошо

Собрать календарь → инвентаризировать каждый слот автозаполнения по отдельности → загрузить недостающие фото → подтвердить все значения полей с владельцем → затем генерировать параллельно, с заранее заполненными полями.

---

## Камень: передача неверных значений в поля изображений автозаполнения

**Почему важно:** когда поле изображения получает пустую строку, путь к
файлу, URL или устаревший/неверный `asset_id`, `~~дизайн` не выдаёт ошибку —
он молча рендерит дефолтную заглушку шаблона (например, обобщённый пейзаж с
облаками и холмами). Макет выглядит «готовым», но главное изображение —
сток, а не фото товара владельца.

### ✗ Плохо

Загрузить фото товара → не дождаться завершения задачи загрузки → передать `job_id` (а не `asset.id`) в данные автозаполнения → дизайн рендерится с заглушкой-пейзажем вместо фото.

### ✓ Хорошо

Загрузить фото через `POST /asset-uploads` → опрашивать `GET /asset-uploads/{job_id}` до `status == "success"` → извлечь `asset.id` из ответа → передать именно этот ID в поле изображения. Проверить, что у каждого поля изображения есть подтверждённый `asset.id`, до вызова автозаполнения.

## Камень: встраивание временных URL превью в markdown

**Почему важно:** ответ автозаполнения возвращает короткоживущие
аутентифицированные URL CDN. Они истекают за минуты. Если встроить их как
markdown-картинки (`![](.../...)`), к моменту показа сообщения они
отрендерятся как битые заглушки «Показать изображение».

### ✗ Плохо

Сгенерировать 8 макетов → вручную встроить каждый временный URL превью в ответ как `![Hero 1](https://...)` → все картинки показываются битыми серыми заглушками, потому что URL истекли до рендера.

### ✓ Хорошо

Либо дайте нативной карусели Cowork отрендерить результат инструмента автоматически, либо экспортируйте каждый дизайн через `POST /exports` и встройте **постоянный** URL экспорта. Финальный запасной вариант: ссылка на постоянный URL редактирования дизайна. Никогда не встраивайте повторно временные превью-URL.

## Камень: перечисление макетов по одному отдельными блоками-картинками

**Почему важно:** когда Claude перечисляет каждый макет отдельным блоком
(«Hero 1 / [картинка] / Hero 2 / [картинка]»), картинки рендерятся битыми
серыми заглушками «Показать изображение». Показ всех макетов вместе одним
сгруппированным пакетом запускает карусель Cowork, которая рендерится верно.

### ✗ Плохо

```
Hero 1 — Портфолио из нескольких объектов
[Показать изображение]
Открыть в ~~дизайне: https://.../d/xxx

Hero 2 — Кейс прибрежного рынка
[Показать изображение]
Открыть в ~~дизайне: https://.../d/yyy
```

### ✓ Хорошо

Показать все макеты в одном сгруппированном ответе, чтобы они отрендерились каруселью. Под каруселью — одна подсказка:

```
Вот 4 hero-макета для недели 1. Пролистайте карусель выше и скажите,
какой оставить.
```

## Камень: генерация макетов последовательно вместо параллельной

**Почему важно:** генерация по одному значит, что владелец ждёт N
оборотов вместо одного. Запускайте все запросы генерации в пакете сразу и
показывайте выбор после завершения всех задач.

### ✗ Плохо

Сгенерировать макет 1 → ждать → показать → макет 2 → ждать → показать → … × 8. Владелец смотрит медленную капельницу 10 минут.

### ✓ Хорошо

Запустить все макеты текущего пакета сразу → опрашивать все задачи параллельно → по завершении экспортировать каждый → проверить визуально → затем показать.

## Камень: автовыбор брендового шаблона на базовом тарифе

**Почему важно:** на базовых тарифах нет API-доступа к брендовым шаблонам.
Если Claude угадает шаблон по имени и сгенерирует не с тем, владелец получит
макеты в неверных цветах/шрифтах и будет переделывать вручную.

### ✗ Плохо

```
Нашёл шаблон «Квадрат — Синий». Использую его для всех постов в ленту.
```
…за чем следуют 8 макетов в синем, когда бренд — зелёный.

### ✓ Хорошо

```
Нашёл 3 шаблона в вашей библиотеке ~~дизайна для постов в ленту:
1. Квадрат — Синий  (правлен март 2026)
2. Летняя акция — Зелёный  (правлен апр. 2026)
3. Карточка товара  (правлен янв. 2026)

Какой использовать?
```

## Камень: дрейф голоса подписей по длинному календарю

**Почему важно:** при написании 12–20 подписей за один проход тон дрейфует —
ранние подписи следуют голосу брифа, поздние сползают в обобщённый
маркетинговый текст. Владельцы замечают сразу.

### ✗ Плохо

Пост 1: «Наши свечи ручной работы — идеальный летний подарок 🌿» (по брифу: «тёплый, разговорный»)
Пост 10: «Раскройте атмосферу с нашей коллекцией авторских ароматов.» (корпоративный дрейф)

### ✓ Хорошо

Перед написанием зафиксируйте 2–3 голосовых маркера из брифа (например, «непринуждённо, по-дружески, с лёгким юмором»). Перечитывайте первую черновую подпись перед каждой новой. Если владелец отметил рассинхрон голоса на какой-то подписи — перечитайте её и сделайте точкой перекалибровки для остальных черновиков.

## Камень: постановка постов в `~~crm` без проверки, что время публикации в будущем

**Почему важно:** `~~crm` отклоняет посты со временем публикации в прошлом с
ошибкой валидации. Если календарь строился на более раннюю дату, а владелец
ставит в очередь только сейчас, некоторые даты могли пройти.

### ✗ Плохо

Построить календарь 01.04 на июньские посты → владелец согласовал 15.05 → попытаться поставить в очередь без проверки → пост на «30.04 10:00» возвращает ошибку.

### ✓ Хорошо

Перед постановкой в очередь сравните каждое время публикации с текущим временем. Если какая-то дата прошла, покажите: «Пост на 30.04 уже прошёл. Пропустить или перенести на следующую неделю?»

---

## Камень: пропуск маркировки рекламы для платных размещений (РФ)

**Почему важно:** в РФ с 01.09.2022 рекламные креативы для **платных**
размещений (Яндекс Директ, VK Реклама, Telegram Ads, посевы в чужих каналах
за деньги) требуют маркировки: токен через ОРД (оператор рекламных данных) и
передача данных в ЕРИР. Пропуск = риск штрафа для владельца. Органические
посты в собственных каналах бренда маркировки не требуют.

### ✗ Плохо

Поставить платный креатив в очередь / запустить таргет без токена ОРД, не уточнив у владельца статус размещения.

### ✓ Хорошо

До постановки в очередь спросить владельца: «Это размещение платное (таргет/посев за деньги) или органический пост в вашем канале?» Если платное — напомнить, что нужен токен ОРД и передача в ЕРИР, и не ставить креатив в очередь без подтверждённого статуса маркировки. ⚠️
